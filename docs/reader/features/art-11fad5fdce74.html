政府使用開源：七國策略差異與落地分法比較

PMPC公共程式 / 數位基建公私協力

來源： 日本 IPA（獨立行政法人資訊處理推進機構）《政府機關開源軟體（OSS）公開活動國際比較調查報告書》（2026 年 3 月 4 日，最後更新 2026 年 3 月 25 日），含兩場圓桌討論；另參考 FSFE 評論、日本 PLATEAU 專案說明、法國 Circulaire Ayrault 及愛沙尼亞《國家財產法》修正案相關報導。

趨勢與重點

規模差距鮮明，但不只是數量問題

日本 IPA 與 monlon 研究員福地宏行、岡本晉主導了這份調查，從 GitHub 蒐集七個國家共 30,298 個儲存庫的量化資料，以儲存庫數、pull request 數、star 數等多項指標進行比較。報告依規模將七國分為三組：

  • 大規模：英國（約 16,000 個儲存庫、約 190 萬個 PR）、美國（約 10,000 個儲存庫、約 54 萬個 PR）
  • 中規模：法國（約 2,400 個儲存庫、約 24 萬個 PR）
  • 小規模：日本（626 個儲存庫、約 3,800 個 PR）、愛沙尼亞、新加坡、德國

各國發展模式各有特色

報告與圓桌討論描述了各國差異化的發展路徑：

英國

活動約自 2010 年起步，2012 年 Government Digital Service（GDS）成立後儲存庫數量大幅成長。由核心組織建立可複製的運作模式，再橫向擴散至各部會。圓桌討論稱之為「種植園模式」。

  • 貢獻者（參與開發人數）在七國中相對較多
  • 強調可複製的標準做法，而非各機關分頭摸索

美國

2016 年 Federal Source Code Policy 前後 OSS 發布增加，但並非由單一核心組織主導，而是聯邦政府多個機關自主推動，從 NASA 到小型研究機構都納入範圍。

  • star 數與 fork 數在七國中相對突出，顯示外部參與程度較高
  • 制度動能來自讓再利用、分支與外部利害關係人參與成為常態，而非政府自行堆疊程式碼

法國

Circulaire Ayrault 通函（2012 年）要求各部會依指引善用自由軟體，說明自由軟體給予使用、修改與散布的自由，強調平等納入評估，並整理跨部會共同制定使用方向的政策背景。

  • 中規模成果顯示行政部門若有明確政策訊號，開源可逐步成為公共數位服務的常態工具
  • 以行政通函而非法律規定推動，屬於「採用建議」而非「強制義務」

德國

依法要求以公帑開發的軟體必須開源，2021 年 Sovereign Tech Fund 構想前後可見發布量增加。但 GitHub 資料仍將德國歸入小規模群。

  • 擁有「數位主權」、「公共資金、公共程式碼」的政策語彙，但量化活動尚未跟上
  • FSFE 2025 年 8 月評論指出，新政府上任一百天後仍缺乏明確方向：採購改革草案未優先支持自由軟體，Germany Stack 走純國家方案可能忽視歐洲層級合作，預算也看不到對公共部門自由軟體的長期支持

愛沙尼亞

國會通過《國家財產法》修正，規定國家全部或部分擁有財產權的軟體原則上應公開，負責機關需決定是否公開並提供描述；若涉及公共秩序或國家安全則可例外不公開。

  • OSS 活動高度集中在 X-Road 資料交換平台，而非大量個別應用程式
  • 以小型國家規模把資源押在能支撐跨機關資料流通的底層基礎設施

新加坡

同屬小規模群，重點在智慧城市相關領域與提升效率的開發平台，而不是大量公開行政服務應用。

  • 以平台型基礎設施作為政府 OSS 的策略支點
  • 目標是讓 OSS 成為公共數位能力的放大器，而非追求英美式的數量規模

日本

規模小於英美法，但與德國、新加坡、愛沙尼亞同屬一個群組，並已確認有局部活躍的發展。每個儲存庫平均約 6.1 個 PR，是小規模組別中最低，報告解讀為重視嚴謹審查流程與品質管理。

  • 動能集中在國土交通省（MLIT）與國土地理院（GSI）的地圖、GIS 與空間資訊領域
  • PLATEAU 計畫（3D 城市模型）在整合 Unity 等通用工具後下載量大幅提升，凸顯配合既 有使用者生態的重要性
  • 數位廳（2021 年成立）雖儲存庫數量不多（調查當時 7 個），卻獲得 1,043 個 star，顯示外部工程師有高度關注與參與意願

研究指出的方法限制

IPA 報告明確說明，本調查僅限於 GitHub 上的量化分析，其他平台、政府內部的 inner-source 協作活動均未納入。各國 OSS 活動量的增加與政策施行時期在時間上有重疊，但報告承認兩者之間的因果關係尚未釐清。

我的觀察

量化資料雖然悲觀，但不能直接以排名當唯一解

這份報告最重要的貢獻之一，是把「日本什麼都沒做」的籠統印象換成可定位的資料。但研究未涵蓋其他平台與政府內部協作，把「小規模」等同於「落後」或「能力不足」，仍是過度推論。尤其把日本較少的 PR 解釋為重視嚴謹審查，這項因果推論本身仍需要額外證據，讀者不宜直接引用作為評比依據。

政策宣示與實際組織化之間存在明顯落差。

德國是最典型的例子：擁有「公共資金、公共程式碼」的法律義務與主權技術語彙，GitHub 活動量卻仍停留在小規模群。FSFE 的觀察進一步指出，即使新政府上任百天，採購改革與預算支持仍未到位。這提醒我們：政策評估不能只看時間線相近，還要追查預算、人力、採購規則與機關實際執行方式。

公開程式碼只是起點，生態系能否接手才是關鍵。

七國政府儲存庫中約三分之一屬於測試程式碼、文件、CI/CD、SDK 等開發工具，而非完成的應用程式。日本 PLATEAU 支援 Unity、提供 Minecraft 轉換器後使用者群擴大的案例，說明政府開源的價值不在「把原始碼放上網」，而在是否提供讓外部開發者能驗證、再利用、接力維護的完整條件。相反地，日本部分儲存庫獲得大量 star 卻未轉化為跨機關採用或持續貢獻，也說明人氣與協作能力是兩件事。

目錄與中繼資料是常被忽略的治理基礎。

研究團隊指出，GitHub 上政府帳號分散、中繼資料不足，光是蒐集、辨識哪些帳號屬於政府就已相當費力。若缺乏可維護的儲存庫目錄、用途描述與責任歸屬，「開源」的成果在事實上幾乎不可被找到或接手。研究團隊提出設立政府開源專案辦公室（OSPO）的構想，但也坦承 OSPO 若只停在倡議，沒有基礎資料管理任務，作用有限。

規模化策略應先辨識優勢，而非複製大國模式。

圓桌討論第二部分強調，日本政府內部工程能量有限，若只假設政府自己會使用與維護開源成果，策略會失準。更務實的路徑是設計讓受委託廠商能採用政府 OSS、再交付回公共系統的循環，並把已在地理資訊等領域自然生長的「原生種」，轉化為組織可延續的能力，而不是硬要複製 GDS 那樣的強心臟核心組織。

後續建議

一、區分「開源數量」與「開源能力」再做政策比較。

若要把這份報告引入台灣或其他行政脈絡的討論，應先追問：哪些面向是平台基礎設施（愛沙尼亞式），哪些是應用程式堆疊（美國式），哪些是帶動再利用的開發工具（PLATEAU 式）？不同形態的開源活動，對行政效率與外部協作的貢獻大不相同，不宜一律用儲存庫數量來衡量。

二、政策評估應加入採購規則與預算追蹤。

德國與日本的對比提供了現成案例：有開源法律義務不等於有組織化的開源動能。評估各國政策成效時，應同步檢視採購規則是否優先考量自由軟體、預算是否有對應支持、執行機關是否有具體人力配置，而不只是看時間線上的政策與 GitHub 數量是否相近。

三、建立與維護政府 OSS 目錄是優先於推廣的基礎工作。

這份報告的製作過程本身就說明問題：研究團隊花費大量時間逐一辨識政府 GitHub 帳號，才能進行分析。若本地政府或研究機構有意跟進類似調查，或協助政府機關推廣開源，建立含責任歸屬、用途描述與授權資訊的儲存庫目錄，會比辦理倡議活動更能直接降低後續協作的門檻。

四、追蹤 FSFE 評論與德國後續政策動向。

FSFE 的觀察目前以倡議立場為主，需與政府文件、預算規劃或採購改革進展交叉查核，不宜單獨引用。若德國 2025 至 2026 年間有政策更新，值得對照 IPA 報告提出的小規模量化表現，檢驗政策語彙是否真的轉化為可觀察的 GitHub 活動。

觀點

延伸觀點

  • 政府開源沒有單一成功模板
    七國資料顯示，英國偏向核心組織帶動、美國採分散擴張，愛沙尼亞與新加坡則集中投入特定基礎設施。與其複製大國模式，更實際的策略是辨識既有優勢，再設計符合本地行政體系的擴散方式。
  • 公開程式碼只是起點
    研究指出，七國政府儲存庫約三分之一屬於測試程式碼、文件、CI/CD、SDK 等開發工具，而非完成的應用程式。這提醒我們，政府開源的價值不只在「把原始碼放上網」，更在於是否提供足以讓外部參與、驗證與再利用的完整條件。
  • 政府 OSS 目錄本身就是治理基礎
    研究團隊指出，政府 GitHub 帳號分散且中繼資料不足，使蒐集、分類與跨國比較都相當困難。OSPO 的基礎任務不應只停在倡議開源，也應建立可維護的儲存庫目錄、用途描述與責任歸屬，讓成果真正可被找到與接手。
  • 政策時間重疊，不等於政策造成成長
    英國 GDS、美國 Federal Source Code Policy、法國 Circulaire Ayrault 等政策或組織成立前後，都出現 OSS 公開量增加，但研究本身承認因果關係尚不明確。政策評估不能只看時間線相近，還要追查預算、人力、採購規則與機關執行方式。
  • 量化比較能破除印象，也可能製造新誤讀
    GitHub 資料有助修正「日本什麼都沒做」的直覺式悲觀，但研究未涵蓋其他平台、政府內部協作與 inner-source，因此不能視為各國行政 OSS 能力排名。報告把日本較少的 pull request 解釋為重視嚴謹審查與品質控管，這項因果推論仍需要額外證據。
  • 人氣不等於協作能力
    日本部分政府儲存庫獲得不少 star，可能反映外界關注，卻尚未明顯轉化為跨機關採用或持續貢獻。star、fork、pull request 各自衡量不同互動，不能把單一數字直接當成開源生態系成熟度。
  • 配合既有工具，比創造專有入口更能擴散
    圓桌討論以 PLATEAU 支援 Unity、Minecraft 等通用工具為例，認為接入既有使用環境有助擴大使用者群。這是來源提出的案例與判斷，實際下載成長幅度仍需要出處，但方向上凸顯互通性比要求使用者遷就政府規格更重要。
  • 從「原生種」長成公共基礎設施
    日本的地圖、GIS 與 PLATEAU 並非源自完整的國家 OSS 策略，而是由第一線需求與特定人才推動。這證明實務需求能孕育開源成果，但若不能跨機關採用並轉化為組織資產，成功仍可能隨人員異動而中斷。
