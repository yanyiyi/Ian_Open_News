# 兩條編輯審查鏈

兩條鏈使用同一個骨架，但判斷標準不同。

## 共用骨架

1. `news-scout` 找題。
2. `angle-strategist` 提出切角。
3. `source-research` 補一手來源與既有資料。
4. 主 session 起草。
5. `structure-editor`、`line-editor`、`target-reader` 平行審稿。
6. 主 session 整理採納清單並改稿。
7. `fact-checker` 查核。
8. PR 合併後更新資料庫狀態。

## 鏈 A：數位人文與在地知識建構

### 找題標準

- 是否與地方知識、文化記憶、數位典藏、公共史學、社群共筆有關。
- 是否能看見人、組織、地方與資料之間的關係。
- 是否涉及資料庫、平台、策展、教育、共創、知識近用。

### 切角問題

- 這則資料呈現了哪一種地方知識建構方法？
- 誰在生產知識，誰被代表，誰能近用？
- 數位工具是放大地方聲音，還是只把地方變成素材？
- 有沒有可連到 OCF、開放文化、公共資料或社群治理的角度？

### 審查清單

- 保留來源、作者、組織、地點、時間與原文連結。
- 避免把地方、族群、社區敘事扁平化。
- 摘要要分清楚「原文說什麼」和「我們的觀察」。
- 若涉及敏感族群、地方衝突、身分資料，需額外標記風險。

## 鏈 B：開放科技與開放產業發展

### 找題標準

- 是否與開源、開放資料、資料治理、標準、授權、公共數位基礎建設有關。
- 是否能看出政策、產業、社群或技術趨勢。
- 是否值得追蹤後續法規、產品、標準、採購、資安或國際組織動向。

### 切角問題

- 這是技術新聞、政策轉向、產業案例，還是社群治理問題？
- 台灣或 OCF 可以從中學到什麼？
- 有沒有開放授權、資料可用性、標準互通、供應鏈安全等關鍵點？
- 需要追哪些一手文件、法規、標準或 GitHub repo？

### 審查清單

- 技術、法規、授權、標準名稱要可查證。
- 避免把 vendor marketing 當成事實。
- 明確標出「新聞事實」、「推論」、「建議」。
- 對產業機會要寫出限制、風險與資料來源。

## PR review 順序

1. 先看 `track`、`status`、`source_id` 與 `url` 是否正確。
2. 再看摘要是否忠於來源。
3. 接著看切角是否符合主線。
4. 最後查核可被驗證的宣稱。

若三審意見互相衝突，由 PR author 在留言中寫採納/不採納理由。
