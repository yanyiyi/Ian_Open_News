多個 agents 幫你
把文章整理好

一條從「找題」到「上線」的內容生產線。把寫作風格、查證、觀點這些每次都會漏掉的事，交給固定角色顧好。

● Claude Code subagents
● 寫作風格規則
● rules vs agents
場景：anoni.net 文章發佈流程
實證：PR #102 已跑過一輪
為什麼要做這件事

發一篇文，要顧的細節太多

寫完當下覺得沒問題，上線後才發現漏了。最常漏的就是這三類。把它們交給固定角色，就不靠記憶。

✍️

寫作風格

語氣不一致、用詞跑掉、雙語版本對不上。每次都要重新提醒自己怎麼寫。

→ 寫作風格規則　+　line-editor
🔍

查證

數字、案例、技術細節有沒有講錯。台下剛好坐著懂的人，一個錯就被抓到。

→ fact-checker
🎯

觀點

切角夠不夠獨特、論證主線清不清楚、讀者看不看得懂、買不買單。

→ angle-strategist · structure-editor · target-reader
先講清楚一個觀念，免得方向錯掉

subagent 不會真的互相對話

✗ 常見誤解
幾個 agent 像開會一樣，彼此即時喊話、辯論，最後生出結論。
✓ 實際運作
每個 agent 跑在自己獨立的 context，做完把結果交回主 session。由主 session 當協調者統整，再決定下一步。

「討論」發生在主 session 的統整層，agent 本身不會互相喊話。真正讓角色彼此辯論的是 Agent Teams，token 量大概是單線的七倍，校一篇文章用它太重。

② 起草

這一步沒有專門的 agent

由主 session 直接起草。它把兩樣東西拼起來：一份固定的寫作風格規則，加上 post-research 備好的料。

寫作風格規則 rule
語氣、用詞、結構的固定守則，常駐自動套用（anoni-docs-style）。
+
post-research 的料
上一步撈好的一手來源與站內該交叉連結的舊文。
→
主 session 起草初稿
docs/zh-TW/blog/posts/xxx.md
依 anoni-docs-style 和剛剛 post-research 的來源，起草 <題目> 到 docs/zh-TW/blog/posts/xxx.md

寫作風格在這一步由 rules 出力：常駐的規則每次自動套，不必另外叫 agent。這也是最後 rules vs agents 那頁的伏筆。

③ 寫完後 · 審稿

三位平行審，主 session 收齊整合

初稿成形後的第一輪。三個互不依賴，所以同時派出去，各跑各的 context 不互相污染。

結構審查
structure-editor
讀完整篇抓出核心主張，檢查每段是否服務主線，哪些順序該調、哪些可刪。只報結構問題，不動字句。
觀點 主線清不清楚
文字潤稿
line-editor
找冗詞、語意含糊、邏輯跳接，檢查語氣前後一致，雙語文件核對中英說的是不是同一件事。針對句子，不重寫整段。
寫作風格 語氣與用詞
讀者代言人
target-reader
站在不熟主題的讀者，標出哪句看不懂、哪個論點不買單、讀完記得什麼。只給讀者反應，誠實、不客氣。
觀點 被不被說服

節奏：三個平行收意見 → 你拍板採納哪些 → 主 session 改寫 → 再丟一次 line-editor 收尾。要串接的話：structure 管大改在前，line 管順句在後。

③ 寫完後 · 查證

fact-checker：最該補的那一位

結構跟文字大致定稿後才查。內容還會大改就先別查，查了白查。

來源與事實查核 fact-checker
列出每一個可被查證的宣稱：數字、日期、國家案例、技術描述、對某專案的描述。逐一判定準確、需要出處、可能有誤、過度宣稱，並附證據與來源連結。
關鍵 要給能查的工具，不然只是憑印象的假查核 tools Read · Grep · WebSearch · WebFetch
差別在這：前三位唯讀就夠，查核必須能上網、或對照 repo 裡的 sources.md，才驗得了外部事實。
實證：PR #102 那篇查了 14 條主張、軟化 1 條，就是這位該做的事。
怎麼叫它們

三種力道，看你多想確保它一定跑

最輕 · 最常用

自然語言點名

用 fact-checker 查 xxx.md

句子裡講出名字，Claude 自己決定要不要派。日常都用這個。

中 · 保證執行

@agent-名稱

@agent-line-editor

打 @ 從選單挑，保證那位一定跑，不讓 Claude 自己選。只想要特定一位審時用。

重 · 整個 session

--agent 旗標

claude --agent news-scout

整個 session 都變成那個角色。適合開一個專門巡題的 session。

平行 vs 串接：獨立工作（三個審稿）平行同時跑，有先後（先 structure 再 line）就串接。
載入有個雷：手寫檔放進 .claude/agents/ 要重開 session 才吃得到。用 /agents 介面建的立即生效。
真正的省力槓桿

把整條包成一句話

每次重打一長串很累。這是會一直重複的事，包成可重複的東西比再多加 agent 有感得多。

scout 挑題→ angle 切角→ research 備料→ 套風格起草→ 三審平行→ fact-check
/new-post　一句帶起整條流程

scout 可拆開：用 claude -p 掛排程定期巡題，把候選自動開成 GitHub issue 進你的題庫。雷達自動掃 → 開 issue → 你挑一個 → 跑 /new-post → 審稿線收尾。

最後比較一下

rules　vs　agents

面向Rules　規則、寫作風格Agents　subagent 角色
運作方式常駐，每次都自動套用按需呼叫，做完才回報
context直接載入主 session各自獨立 context，互不污染
性質被動約束：規定「該怎麼寫」主動執行：去審、去查、去找
適合一致的風格、格式、底線一次性的審查、查證、研究任務
成本幾乎零額外 token每位回報都吃 context，越多越雜
會不會漏永遠都在，不會忘要記得叫，沒叫就不會跑
比喻守則貼在牆上找專人來審一輪

兩者互補。「每次都要、不能漏」的（風格、底線）用 rules 釘死，不靠記憶。「要動腦、要跑一輪」的（查證、切角、審稿）用 agents。anoni 那條流程正是合用：一份寫作風格規則 + 六個審稿與研究 agents。

帶走這幾點

重點回顧

1

發文常漏的寫作風格、查證、觀點，交給固定角色顧，不靠當下的記憶。

2

subagent 不互相對話。各跑各的 context，主 session 當協調者統整。

3

一條從找題到上線的生產線：上游找題備料 → 起草 → 審稿查證。

4

角色不是越多越好。每位回報都吃 context，加到夠用就停。

5

常駐的用 rules，要跑一輪的用 agents，最後包成 /new-post 一句帶起。

範例檔案

agent 檔就是一個帶 YAML frontmatter 的 markdown：frontmatter 設定角色，正文是這個角色的 system prompt。挑四個當範本。

① 上游

議題雷達 news-scout

.claude/agents/news-scout.md
---
name: news-scout
description: 議題雷達。盯匿名網路、數位身分、審查量測、數位人權的近期新聞，回報值得 anoni.net 寫的題目。要找題目時使用。
tools: Read, Grep, Glob, WebSearch, WebFetch
model: sonnet
---

你替 anoni.net 隱私社群盯新聞，讀者是記者、公民團體、開源科技社群。

關注 beat：Tor、Tails、OONI、數位身分與 eID、監控與審查
立法、審查量測、加密貨幣與支付隱私、台灣與 APAC 數位人權。

被指派找題目時：
1. 用 WebSearch 掃指定時間範圍（預設近兩週）各 beat 的新事件
2. 篩掉炒作、純產品發布、跟隱私或匿名無關的
3. 每則候選回報：事件一句話、日期、一手來源連結、
   跟 anoni 的關聯、有沒有在地角度、時效
4. 依「值得寫」程度排序，最多六則

不要寫文章，不要決定切角，只把值得評估的題目攤出來。
用途
想找新題目、定期巡一輪時。
一定要
給它 WebSearch，不然只是憑印象。
叫法
用 news-scout 掃近兩週各 beat
model
sonnet 就好，巡題不必動到 opus。
③ 審稿

文字潤稿 line-editor

.claude/agents/line-editor.md
---
name: line-editor
description: 文字潤稿者。檢查語句通順、用詞精準、語氣一致、中英對照是否對得上。審稿時使用。
tools: Read, Grep, Glob
model: sonnet
---

你是一位文字編輯，把話講清楚、講準是你的本職。

審稿時：
1. 找出冗詞、語意含糊、邏輯跳接的句子
2. 檢查語氣是否前後一致
3. 雙語文件時，核對中英兩版說的是不是同一件事

每個問題給出：原句位置、問題、改寫版本。
不要重寫整段，針對句子。
用途
初稿成形後的審稿輪，三位平行其一。
設定
唯讀（Read、Grep、Glob），只審不改，改寫留給主 session。
對應痛點
寫作風格。
叫法
@agent-line-editor 保證只跑這位。
③ 查核

事實查核 fact-checker

.claude/agents/fact-checker.md
---
name: fact-checker
description: 來源與事實查核者。核對文章裡的事實宣稱、數字、案例、專案描述是否準確、有無出處。審稿時使用。
tools: Read, Grep, Glob, WebSearch, WebFetch
model: sonnet
---

你是一位事實查核編輯，讀者是熟悉 Tor、OONI、數位人權的
專家，任何不準的宣稱都會被抓到。

審稿時：
1. 列出每一個可被查證的宣稱：數字、日期、國家案例、
   技術描述、對某專案或組織的描述
2. 逐一判定：準確、需要出處、可能有誤、過度宣稱
3. 對「可能有誤」用 WebSearch、WebFetch 去核，對照
   引用來源（若 repo 裡有 sources.md）確認沒有曲解

每個問題回報：原句位置與宣稱、判定結果、證據或
來源連結、建議改法。不要潤稿，只管「這句站不站得住」。
時機
結構與文字定稿後才查，內容還會大改先別查。
關鍵
要能上網或對照 sources.md，否則是假查核。
對應痛點
查證。
實證
PR #102 那篇查了 14 條、軟化 1 條。
